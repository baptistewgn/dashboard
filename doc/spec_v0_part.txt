Technical Specification — Macro / FX / Crypto Research Dashboard
1. Purpose
Design and implement a research dashboard for daily monitoring of macro, FX, and crypto markets. The dashboard is not a signal engine by itself; it is a state monitoring and regime classification system used to:
1. track market context,
2. monitor research variables,
3. identify regime transitions,
4. support discretionary and systematic strategy research.
The dashboard should be suitable for:
* daily research review,
* idea generation,
* pre-trade context assessment,
* post-move diagnosis,
* validation of strategy conditionality across regimes.

2. Core Design Principles
2.1 Research-first, not chart-first
The system must prioritize:
* economically meaningful variables,
* consistent transformations,
* comparable horizons,
* regime labeling,
* low cognitive load.
2.2 Hierarchical structure
The dashboard should be organized by decision layer rather than asset class alone.
Recommended top-level layout:
1. Global Macro Liquidity / Rates
2. Risk Regime
3. FX Drivers
4. Crypto Market Structure
5. Crypto Leverage / Positioning
6. Crypto Flows / Adoption
7. Calendar / Event Risk
2.3 State representation
Each panel should expose three levels of information:
* Level 1: latest value
* Level 2: recent change (1d / 5d / 21d)
* Level 3: normalized state (z-score / percentile / regime label)
2.4 Decision support rules
Every metric included must answer at least one of these questions:
* Is liquidity tightening or easing?
* Is the environment risk-on or risk-off?
* Are rate differentials supportive for FX trends?
* Is crypto leverage crowded or unwinding?
* Are flows supportive or distributive?
* Is a strategy regime changing?

3. Dashboard Objectives
3.1 Primary objectives
* Provide a daily overview of macro and cross-asset conditions.
* Monitor core explanatory variables for FX and crypto.
* Identify crowding, stress, and trend persistence.
* Detect changes in conditions under which strategies historically work or fail.
3.2 Secondary objectives
* Serve as a standardized research workspace.
* Improve repeatability of market review.
* Reduce noise from ad hoc chart checking.
* Create structured inputs for downstream backtests and logging.

4. Target Users
Primary user
* discretionary macro / crypto researcher,
* systematic strategy researcher,
* quant PM / analyst.
User assumptions
* understands rates, vol, spreads, carry, term structure, leverage, and basic on-chain metrics,
* wants fast daily interpretation,
* values statistically consistent signal representations.

5. Functional Scope
In scope
* multi-asset monitoring,
* daily and intraday-refresh market state display,
* normalized indicators,
* event highlighting,
* regime classifiers,
* watchlist and cross-sectional comparison,
* exportable snapshots.
Out of scope (v1)
* automated execution,
* order placement,
* complex forecasting models,
* full portfolio management suite,
* alert optimization via ML.

8. Optimized Metric Set
8.1 Global Macro / Liquidity
Keep from proposal
* US 2Y yield
* US 10Y yield
* US 30Y yield
* yield curve slope
* SPX
* Euro Stoxx 50
* Gold
* Brent
* VIX
* VSTOXX
Additions
* DXY
* US 10Y real yield (or 10Y TIPS yield)
* MOVE index
* US 2s10s and 5s30s slope
* EU 2Y / 10Y yields
* credit proxy if available (CDX IG / HY or equivalent ETF proxy)
Why these additions matter
* DXY proxies global dollar tightness.
* Real yields are more informative than nominal yields for duration-sensitive and risk assets.
* MOVE captures bond volatility, often leading cross-asset stress.
* EU rates are required for cleaner FX relative-value interpretation.

8.2 FX-Specific
Keep from proposal
* EURUSD
* EUR crosses vs CHF / GBP / CAD / JPY
* USD crosses vs CHF / GBP / CAD / JPY
* EUR-US yield spread
* ECB policy rate
* Fed policy rate
* EUR inflation
* US inflation
Improve specification
Replace generic “yield spread” with term-specific spreads:
* US 2Y minus DE 2Y
* US 10Y minus DE 10Y
* US 2Y minus EZ 2Y policy proxy if DE unavailable
Additions
* DXY
* broad EUR index and USD index if available
* implied vol for EURUSD
* 1m / 3m risk reversals for EURUSD if available
* market-implied terminal ECB / Fed rate
* next meeting implied cut/hike probability
Why these additions matter
Spot FX alone is incomplete. FX requires monitoring:
* rate differential level,
* rate differential change,
* expected policy path,
* volatility pricing,
* event risk concentration.

9. Visualization Specification
9.1 Card format
Each card should contain:
* metric name,
* latest value,
* 1d / 5d / 21d change,
* 63d z-score,
* mini sparkline,
* regime tag.
Example:
BTC | 84,250 | +2.1% / +7.8% / +14.3% | z63=1.7 | Trend+ / Crowded
9.2 Chart windows
Recommended defaults:
* 3m: tactical monitoring
* 1y: regime context
* 3y: structural perspective
Do not use only 3m windows. A 3m view is useful for short-term context but often hides whether a move is statistically extreme versus 1y or 3y history.
9.3 Transformations shown on charts
Support overlays for:
* raw level,
* rolling z-score,
* rolling percentile,
* moving average bands,
* drawdown from local peak,
* event markers.
9.4 Color logic
Use a restrained color system:
* green = supportive / easing / bullish,
* red = tightening / stress / bearish,
* amber = transitional / mixed,
* blue = neutral informational state.
Avoid using price-up = green universally. For yields and volatility, sign interpretation is context dependent.

10. Standardized Transformations
10.1 Return horizons
Use your proposed convention broadly:
* 1d change
* 5d change
* 21d change
Also add:
* 63d change for medium-term trend,
* YTD change for dashboard snapshots.
10.2 Normalization
For non-price variables, compute:
Rolling z-score
z_t = (x_t - mean_{t-w}) / std_{t-w}
Recommended windows:
* 63d tactical
* 252d contextual
Percentile rank
Useful when distributions are skewed or fat-tailed:pct_t = rank(x_t within lookback window)
Rate of change
For OI, stablecoin cap, ETF flows:
* 1d delta
* 7d cumulative delta
* 30d cumulative delta
10.3 Smoothing rules
Apply light smoothing for noisy series only:
* 3d EMA or 7d EMA for funding and liquidations,
* no smoothing for spot price and major yields in headline values.

11. Regime Classifier Framework
The dashboard should include simple, transparent, rules-based classifiers before attempting ML.
11.1 Global Risk Regime
Inputs
* SPX 21d return
* VIX percentile
* US 10Y real yield change (21d)
* credit spread proxy change
Example rule set
* Risk-On if SPX > 0 over 21d and VIX percentile < 60 and real yields not sharply rising
* Risk-Off if SPX < 0 over 21d and VIX percentile > 75
* Stress if VIX percentile > 90 and credit spreads widening sharply
* Mixed otherwise
Output labels
* Risk-On
* Mixed
* Risk-Off
* Stress

11.2 Liquidity Regime
Inputs
* DXY 21d change
* real yield 21d change
* yield curve slope change
* stablecoin market cap 30d change (crypto subcomponent)
Rule intuition
* rising DXY + rising real yields = liquidity tightening
* falling DXY + falling real yields = liquidity easing
Output labels
* Easing
* Neutral
* Tightening
* Acute Tightening

11.3 FX EURUSD Regime Classifier
Inputs
* US-DE 2Y spread level and 21d change
* EURUSD 21d return
* EURUSD realized vol
* implied ECB / Fed pricing change
Rule interpretation
* wider US-DE 2Y spread in favor of USD = EURUSD bearish macro backdrop
* narrowing spread = EURUSD supportive
Output labels
* EUR Supportive
* USD Supportive
* Range / Neutral
* Event-Driven

11.4 Crypto Trend Regime
Inputs
* BTC 21d and 63d return
* total market cap 21d return
* BTC dominance 21d change
* realized vol percentile
Rule set example
* Bull Trend: BTC > 0 on 21d and 63d, total cap > 0, vol not in extreme stress
* Bear Trend: BTC < 0 on 21d and 63d, total cap < 0
* Rotation: BTC up modestly, dominance falling, alts outperforming
* Defensive BTC: BTC flat/up, dominance rising, alts weak
Output labels
* Bull Trend
* Bear Trend
* Alt Rotation
* Defensive BTC
* Choppy

11.5 Crypto Leverage Regime
Inputs
* BTC OI z-score
* BTC funding z-score
* basis percentile
* liquidation imbalance
Rule logic
* high OI + high funding + rich basis = crowded longs
* high OI + deeply negative funding = crowded shorts
* falling OI after extremes = unwind
Output labels
* Crowded Longs
* Crowded Shorts
* Unwind
* Clean / Neutral
* Squeeze Risk

11.6 Crypto Flow Regime
Inputs
* ETF net flows 5d / 21d
* exchange netflows
* stablecoin supply 30d delta
* BTC / ETH exchange balances
Output labels
* Accumulation
* Distribution
* Passive Supportive
* Weak Demand

11.7 Strategy Suitability Overlay
This is the most useful practical layer.
Map regimes to research conditions:
* Momentum: favorable in persistent trend + moderate vol
* Mean reversion: favorable in choppy / range + high local stretch
* Funding reversal: favorable in crowded leverage regimes
* Basis / carry: favorable in stable leverage demand and low stress
* Breakout strategies: favorable after volatility compression with macro catalyst risk
This layer should remain descriptive, not prescriptive.

12. Event and Calendar Layer
The dashboard should support a compact event strip showing:
* next ECB meeting
* next FOMC meeting
* CPI dates (US / EZ)
* NFP
* major crypto ETF decision / known catalyst dates if tracked
Each event should include:
* date/time,
* consensus / implied probability where available,
* impact flag,
* countdown.
Recommended tags:
* High impact
* Medium impact
* Crypto-specific
* Central bank
* Inflation

13. Data Requirements
13.1 Frequency by module
* Macro / rates / equities / commodities: daily, with intraday optional
* FX spot: daily close plus intraday optional
* Crypto: hourly and daily aggregation recommended
* Funding / OI / liquidations: hourly preferred, daily mandatory for dashboard rollups
* ETF flows: daily
* Stablecoin cap: daily
13.2 Alignment rules
* Store all timestamps in UTC.
* Render user-facing timestamps in local timezone if desired.
* Daily close convention must be explicit.
* Crypto should use rolling 24h logic where relevant; macro/FX should use official market close convention.
13.3 Data quality rules
Every series must have:
* source identifier,
* last-update timestamp,
* stale-data threshold,
* missing-data fallback rule.
Quality flags:
* OK
* Delayed
* Partial
* Missing
* Suspect
